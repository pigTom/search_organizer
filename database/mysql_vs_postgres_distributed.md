## 架构差异

```
┌─────────────────────┬──────────────────────┬──────────────────────┐
│        维度          │       MySQL          │     PostgreSQL       │
├─────────────────────┼──────────────────────┼──────────────────────┤
│  进程模型            │ 多线程（轻量）       │ 多进程（较重）        │
│  连接开销            │ 低（线程约 256KB）   │ 高（进程约 5-10MB）   │
│  简单查询吞吐        │ 更高                 │ 略低                  │
│  复杂查询优化        │ 较弱                 │ 更强                  │
│  MVCC 实现           │ Undo Log 回滚段      │ 堆表多版本 + VACUUM   │
│  写放大              │ 较低                 │ 较高（需要 VACUUM）   │
└─────────────────────┴──────────────────────┴──────────────────────┘
```

### 进程模型的影响

- **MySQL 多线程模型**：线程间共享内存，创建和切换开销小，适合大量短连接
- **PostgreSQL 多进程模型**：每个连接独立进程，隔离性好但内存占用高，高并发时必须配合连接池（如 PgBouncer）

### MVCC 实现差异

- **MySQL**：原地更新 + Undo Log 保存旧版本，通过回滚指针形成版本链
- **PostgreSQL**：追加写入新版本，旧版本留在堆表中，依赖 VACUUM 清理死元组

## 分布式读写分离场景

### MySQL（主从复制）

```
优势：
├─ 原生主从复制成熟稳定，部署简单
├─ binlog 复制生态丰富（Canal、Maxwell、Debezium）
├─ 半同步复制（Semi-Sync）可控制数据安全
├─ Group Replication 提供多主写入能力
└─ 简单 OLTP 场景下吞吐量更高

劣势：
├─ 异步复制有延迟，可能读到旧数据
├─ 主从切换需要外部工具（MHA、Orchestrator）
└─ 逻辑复制能力较弱（MySQL 8.0 有改善）

典型方案：
├─ MySQL + ProxySQL / MaxScale（读写分离中间件）
├─ MySQL Group Replication（MGR）
└─ MySQL InnoDB Cluster
```

### PostgreSQL（流复制）

```
优势：
├─ 流复制（Streaming Replication）延迟极低
├─ 逻辑复制（Logical Replication）灵活，支持部分表复制
├─ 同步复制可配置多种级别（remote_write / remote_apply）
├─ 从库可执行复杂查询，不影响主库
└─ 内置故障转移支持（pg_rewind）

劣势：
├─ 连接数受进程模型限制，需要连接池（PgBouncer）
├─ VACUUM 可能影响复制延迟
└─ 生态工具不如 MySQL 丰富

典型方案：
├─ PostgreSQL + Patroni（高可用）
├─ PostgreSQL + PgBouncer（连接池）
└─ PostgreSQL + Citus（分布式扩展）
```

## 关键性能指标对比

### 高并发短事务（OLTP 核心场景）

```
MySQL 更优：
├─ 线程模型在高并发下内存占用远低于 PG
├─ 简单主键查询 MySQL 快 10-30%
├─ 连接创建/销毁成本低
└─ 互联网公司大规模验证（阿里、美团、字节）

基准参考（sysbench oltp_read_write，16核64G）：
├─ MySQL 8.0:  ~15,000 TPS（128并发）
├─ PostgreSQL 16: ~12,000 TPS（128并发，配合PgBouncer）
└─ 差距随并发数增大而增大
```

### 复杂分析查询（OLAP / 混合负载）

```
PostgreSQL 更优：
├─ 查询优化器更成熟（支持 Hash Join、Merge Join、并行查询）
├─ MySQL 直到 8.0 才支持 Hash Join
├─ PG 并行查询能力更强（Parallel Seq Scan, Parallel Hash Join）
├─ 支持丰富的数据类型（JSONB、数组、范围类型）
├─ 支持物化视图、CTE、窗口函数（MySQL 8.0 才跟上）
└─ 对复杂 SQL 的执行效率优势明显

基准参考（TPC-H 10GB，复杂分析查询）：
├─ PostgreSQL 通常快 2-5 倍
└─ 部分查询差距可达 10 倍以上
```

### 写入密集场景

```
┌──────────────────┬─────────────────────┬──────────────────────┐
│                   │      MySQL          │    PostgreSQL        │
├──────────────────┼─────────────────────┼──────────────────────┤
│ 单行写入吞吐     │ 更高                │ 略低                 │
│ 批量写入          │ 高                  │ COPY 命令非常快      │
│ 写放大            │ 低（in-place更新）  │ 高（追加写+VACUUM）  │
│ 长期运行稳定性    │ 稳定                │ 需要调优 VACUUM      │
│ SSD 磨损          │ 更少                │ 写放大导致磨损更多   │
└──────────────────┴─────────────────────┴──────────────────────┘
```

#### PostgreSQL VACUUM 问题

- 更新不是原地修改，而是插入新版本 + 标记旧版本为死元组
- VACUUM 负责清理死元组，回收空间
- 高写入场景下 VACUUM 跟不上会导致表膨胀
- autovacuum 调优是 PG 运维的核心挑战
- MySQL 的 Undo Log 方式在这方面天然更优

## 分布式场景选型建议

```
选 MySQL 的场景：
├─ 互联网高并发 OLTP（电商、社交、支付）
├─ 连接数多、查询简单
├─ 团队 MySQL 经验丰富
├─ 已有成熟的 MySQL 中间件体系
└─ 需要 TiDB 生态（MySQL 协议兼容）

选 PostgreSQL 的场景：
├─ 查询复杂度高（报表、分析、数据仓库）
├─ 需要丰富的数据类型（GIS、JSON、全文检索）
├─ 需要强一致性和完整的 SQL 标准支持
├─ 考虑 NewSQL（CockroachDB、YugabyteDB 兼容 PG）
└─ 混合 OLTP+OLAP（HTAP）负载
```

### 总结

> **MySQL 像跑车——简单场景下极快；PostgreSQL 像越野车——复杂地形下更可靠。**
> 分布式场景下，两者原生能力都不够，最终比的是各自生态中间件（Vitess/TiDB vs Citus/CockroachDB）的成熟度和适配度。

| 场景 | 推荐 | 原因 |
|------|------|------|
| 高并发简单 OLTP | MySQL | 线程模型轻量，吞吐更高 |
| 复杂查询 / OLAP | PostgreSQL | 优化器更强，并行查询能力好 |
| 写入密集 | MySQL | 无 VACUUM 开销，写放大更低 |
| 分布式强一致 | PostgreSQL 生态 | CockroachDB/YugabyteDB 更成熟 |
| 互联网标准架构 | MySQL | 生态丰富，运维经验成熟 |

---

*本文档对比基于 MySQL 8.0+ 与 PostgreSQL 15/16，具体性能因硬件、配置和负载而异。*
